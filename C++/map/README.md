# Условие задачи: Allocator-Aware UnorderedMap

В этой задаче Вам предлагается реализовать упрощенный аналог одного из популярных контейнеров STL — `unordered_map`. 

Вам очень пригодится `List`, написанный в предыдущей задаче. Использовать `std::list` нельзя.

На этот раз нужно будет правильно поддержать не только итераторы и аллокаторы, но еще и **move-семантику**. Все классы, которые вы используете для своей реализации (например, `List`), также должны корректно поддерживать move-семантику. Если ранее `List` ее не поддерживал, вам нужно будет её дописать.

---

## Требования к классу
Напишите шаблонный класс `UnorderedMap`:

```cpp
template<
    class Key, 
    class Value, 
    class Hash = std::hash<Key>, 
    class Equal = std::equal_to<Key>, 
    class Alloc = std::allocator<std::pair<const Key, Value>>
>
class UnorderedMap;
```

Внутри `UnorderedMap` публично определите тип `NodeType` как `std::pair<const Key, Value>`. (В `std::unordered_map` аналогичный тип называется `value_type`).

### Требования к типам Key и Value
Типы `Key` и `Value` **не обязаны** иметь конструктор по умолчанию, копирующий конструктор и копирующий оператор присваивания. Для создания `UnorderedMap` с ними должно быть достаточно, чтобы они имели лишь move-конструктор и move-оператор присваивания.

---

## Функциональность

### Базовые операции и методы доступа
* **Конструкторы:** (в т.ч. копирования и перемещения), деструктор, копирующий и перемещающий операторы присваивания.
* **Доступ к элементам (в среднем за O(1)):**
  * Оператор `[]`: определен только для неконстантных объектов. Создает `Value` по умолчанию, если ключа не было. Если в метод `[]` передана rvalue-ссылка на ключ и ключа в мэпе нет, нужно эффективно создать `Node` — не копировать ключ, а **перемещать**.
  * `at(key)`: кидает исключение `std::out_of_range`, если ключа нет.
* **Размер:** `size()` (за гарантированное O(1)) и `empty()`.
* **Поиск (в среднем за O(1)):** `find(key)`, возвращающий итератор на найденный элемент или `end()`.

### Итераторы
* Внутренние типы `iterator` и `const_iterator`, удовлетворяющие требованиям `ForwardIterator`. Разыменование константного итератора не должно позволять менять объект под ним.
* Методы `begin()`, `end()`, `cbegin()`, `cend()`.
* **Ограничение производительности:** Код `for (auto it = m.begin(); it != m.end(); ++it);` должен позволять пройтись по всем элементам за время, линейное от **количества этих элементов** (а не от размера хеш-таблицы).
* Инкрементация любого итератора должна происходить за **гарантированное O(1)**.

### Модификация контейнера
* **insert** (в среднем за O(1)):
  * `insert(NodeType&&)` / `insert(const NodeType&)` — принимает ссылку на `NodeType`, возвращает `std::pair<iterator, bool>`. Должен эффективно обрабатывать lvalue и rvalue (не копировать, если передано rvalue).
  * `insert(InputIt first, InputIt last)` — вставляет элементы из диапазона. Если разыменование итератора дает rvalue-ссылку (например, при `std::move_iterator`), элементы должны вставляться **перемещением**, а не копированием.
* **emplace(Args&&... args):** Самостоятельно конструирует `NodeType` от переданных аргументов (без их лишнего копирования) и вставляет в контейнер. Возвращаемый тип — `std::pair<iterator, bool>`.
* **erase:** Удаление одного элемента по итератору `erase(iterator)` или диапазона `erase(first, last)`. Удаление одного элемента должно работать за O(1) в среднем. Итераторы на элементы, которые не были удалены, не должны инвалидироваться.
* **swap(other):** Должен работать за O(1). При `swap` итераторы в обоих `UnorderedMap` **не должны инвалидироваться**, а должны продолжать указывать на те же элементы, куда указывали ранее.

### Управление хеш-таблицей
* Методы `reserve`, `load_factor` и `max_load_factor`, аналогичные методам STL.

---

## Гарантии безопасности (Exception Safety & Invalidation)
* Вставка и удаление любого количества элементов **не должны инвалидировать указатели и ссылки** на остальные элементы (на `NodeType`).
* Однако вставка может инвалидировать итераторы, если во время нее произошел rehash.
* Методы `insert` и `emplace`, вставляющие одиночные элементы, должны быть **строго безопасны** относительно исключений (кроме случая, когда исключение кидается самой хеш-функцией).

---

## Allocator-Aware требования и Empty Base Optimization
* Никаких прямых обращений к `new`/`delete` быть не должно, всё должно происходить через `std::allocator_traits`.
* Во время конструирования и присваивания должны правильно использоваться свойства аллокаторов:
  * `select_on_container_copy_construction`
  * `propagate_on_container_copy_assignment`
  * `propagate_on_container_move_assignment`
  * `propagate_on_container_swap`
* **EBO (Empty Base Optimization):** Если аллокатор является структурой без полей, он не должен увеличивать `sizeof` объекта `UnorderedMap` (тривиальный аллокатор как поле должен занимать 0 байт). Это же требование распространяется на объекты `Hash` и `Equal`.
